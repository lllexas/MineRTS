# Bug 报告: 拆除建筑后 NavMesh 未刷新，寻路单位未重新寻路

## 基本信息
- **报告日期**: 2026-02-27
- **优先级**: 中高 (影响核心玩法)
- **状态**: 待修复
- **发现者**: 用户
- **记录者**: 小喵 (Claude Code)

## 问题描述
当玩家拆除建筑后，该建筑原本占据的区域应该变为可通行区域，NavMesh 应该相应刷新。但当前存在以下问题：

1. **NavMesh 未刷新**: 拆除建筑后，对应的网格区域没有被正确标记为可通行
2. **寻路单位未重新寻路**: 正在向该区域移动的单位不会重新计算路径，可能卡住或尝试穿过不可通行区域

## 重现步骤
1. 进入任意关卡（例如 `stage_01`）
2. 建造一个建筑（例如矿场、发电站）
3. 命令一个单位移动到建筑另一侧
4. 在单位移动过程中拆除该建筑
5. 观察单位行为和网格可通行性

## 预期行为
1. 建筑拆除后，其占据的网格立即变为可通行
2. NavMesh 自动重建该区域的路网拓扑
3. 正在寻路的单位检测到路径变化，重新计算最优路径
4. 单位顺利通过原建筑区域

## 实际行为
1. 建筑视觉上消失，但网格仍被标记为占据（或 NavMesh 未更新）
2. 单位继续沿原路径移动，在建筑原位置卡住或绕行失败
3. 需要手动重新下达移动指令才能继续前进

## 相关代码位置

### 1. 建筑拆除逻辑 (`EntitySystem.cs:271-351`)
```csharp
// 第 298-306 行：建筑拆除时的 NavMesh 重建
if ((unitType & UnitType.Building) != 0)
{
    int startX = logicalPos.x - (size.x - 1) / 2;
    int startY = logicalPos.y - (size.y - 1) / 2;
    RectInt destroyArea = new RectInt(startX, startY, size.x, size.y);

    // 重新剖分这块区域，它会自动把原来的"洞"补上，并和邻居合并成大矩形
    GridSystem.Instance.RebuildNavMesh(destroyArea);
}
```

### 2. NavMesh 重建逻辑 (`GridSystem.cs:706-752`)
- `RebuildNavMesh(RectInt dirtyArea)` 方法负责局部区域重建
- 逻辑包括：寻找受影响节点、确定重绘包围盒、贪婪矩形剖分、重建邻接关系

### 3. 寻路系统 (`PathfindingSystem.cs`)
- 需要检查是否有监听 NavMesh 变化的机制
- 单位路径是否在 NavMesh 更新后重新验证

## 可能的原因分析

### 原因 1: 建筑类型判断问题
- `(unitType & UnitType.Building) != 0` 可能未正确识别所有建筑类型
- 某些建筑可能有复合类型标记（如 `Building | Resource`）

### 原因 2: RebuildNavMesh 实现缺陷
- `RebuildNavMesh` 可能未正确处理"移除障碍"的情况
- 矩形剖分算法可能在某些边缘情况下失败
- 邻接关系重建可能不完整

### 原因 3: 寻路系统未响应变化
- `PathfindingSystem` 可能没有订阅 NavMesh 更新事件
- 单位路径可能缓存了旧的通行性信息
- 没有定期重新验证路径有效性的机制

### 原因 4: 网格占据清理不彻底
- `GridSystem.Instance.ClearOccupantRect` 可能没有正确清理所有相关网格
- 网格索引计算可能存在偏差

## 影响范围
- **游戏体验**: 玩家需要频繁手动重新寻路，影响操作流畅性
- **战术层面**: 动态改变地形是 RTS 游戏的重要策略，此 bug 限制了战术深度
- **性能影响**: 如果大量单位卡住，可能影响游戏性能
